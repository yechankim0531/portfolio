## Overview

TaskFlow started as a weekend experiment to see whether a genuinely fast, keyboard-first task board was possible in the browser. It grew into a full product with workspaces, team invites, and a lightweight analytics layer.

The core insight driving the design is that **most project-management tools are optimized for managers, not makers**. TaskFlow flips that: the default view is a focused personal queue, and the board is one click away when you need the big picture.

## The problem

Existing tools like Jira and Linear are powerful but slow. Opening a ticket, updating a status, or moving a card between columns often requires three or four interactions. For engineers doing this fifty times a day, the friction adds up.

## Architecture

The app is a Next.js 14 monolith with an App Router front-end and an API layer that speaks directly to PostgreSQL via `pg`. WebSocket state is managed by a small Socket.io server running as a Next.js custom server.

```
client → Next.js API route → PostgreSQL (RLS enforced)
                          ↘ Socket.io broadcast → other clients
```

One deliberate constraint: **no ORM**. Raw SQL is easier to reason about for complex permission queries, and PostgreSQL's query planner handles the rest.

## What I learned

Building the real-time sync layer exposed a class of bugs I hadn't encountered before: events arriving out of order, duplicate broadcasts when a client reconnects mid-operation, and stale reads racing against in-flight mutations.

The solution was a simple version vector attached to each board. Clients reject events with a lower version than their local state, and request a full snapshot when they detect a gap.
